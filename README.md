# 🍔 大学城美食地图 (Campus Food Map) - MVP 原型

> **基于 AI 辅助工具（Cursor）在 48 小时内完成的本地生活餐饮决策 Web App 体验原型。**

## 🎯 项目背景与核心痛点

本项目旨在解决大学城（3-5所高校聚集地）学生群体在日常就餐时面临的决策困境。通过前期调研，梳理出以下三大核心痛点：
1. **决策疲劳**：传统本地生活平台信息过载，学生由于预算有限害怕“踩雷”，导致反复吃同一家店，缺乏探索欲。
2. **群体决策摩擦**：宿舍或社团聚餐时，“随便”、“都行”的表态往往导致决策效率低下。
3. **校园特定物理限制**：外卖往往无法送达宿舍楼上，高峰期存在严重的“最后100米”拥挤和防丢焦虑。

## 💡 产品解法与核心功能设计

为了在 MVP 阶段快速验证核心假设，本应用设计了 5 个核心模块，重点优化用户的“发现、决策与互动”体验：

* **🌍 发现 (Discover)**：
  * **场景地图化**：采用 2.5D 视图取代传统列表，直观呈现大学城商业聚落。
  * **顺路代拿（轻社交互动）**：通过顶部的实时滚动信息，营造校园特有的“互助代拿”与社交活跃氛围。
* **🏆 榜单 (Playlists)**：
  * **情绪化导航**：打破传统的“按菜系”分类，采用“早八救赎”、“期末破防”、“减脂专区”等契合学生心智的场景化标签。
* **🎲 转盘 (Decision)**：
  * **消除决策阻力**：引入随机抽取引擎与丝滑的动画动效，将“选哪家”变成一种低门槛的游戏化体验。
  * **防踩雷试错**：强化“15元特供试错套餐”与“卫生评级”的 UI 展现，降低学生的试错心理成本。
* **🔥 组局 (Match)**：
  * **异步投票工具**：采用类似 Tinder 的左右滑动卡片交互，高效解决宿舍聚餐“吃什么”的群体共识问题。
* **👤 我的 (Profile)**：
  * **轻量级二元评价**：用“比上次好吃吗？”的简单判断代替 1-5 星长文本打分，降低评价门槛，逐步建立个人“美食天梯榜”。

## ⏳ 48小时开发策略与工程取舍 (Trade-offs)

鉴于测评对开发时间（48小时）的硬性要求，我在项目执行中采取了以下敏捷开发策略：

1. **AI 协同编程 (Cursor)**：利用 Claude 3.5 Sonnet 快速生成底层的 Next.js 结构与 Tailwind CSS 样式，将主要精力释放在产品逻辑梳理与交互体验（Framer Motion 动效）的打磨上。
2. **纯前端架构 (Mock Data)**：为了优先保证产品结构和功能设计的完整展现，放弃了复杂的后端数据库搭建。全部采用前端 JSON 静态数据，模拟接口延迟与加载状态。
3. **Fake UI 验证**：对于“微信登录”、“美团外卖跳转”等需要极高接口权限的功能，采用高保真 UI 与 Toast 提示的闭环演示，重点展示业务链路而非底层代码。

## 🚀 未来演进规划 (Roadmap)

如果项目进入后续的真实商业落地阶段，演进计划如下：

* **Phase 1 (冷启动验证)**：核心关注转盘的使用频次与“情绪榜单”的点击转化率，验证用户是否愿意为“降噪后的推荐”买单。
* **Phase 2 (UGC 沉淀)**：接入真实后端（如 Supabase），上线学生 Edu 邮箱认证，沉淀真实的校园二元评价数据，防止商家刷单。
* **Phase 3 (商业化闭环)**：打通真实的 LBS 定位服务，结合校园下午茶等闲时场景，引入“动态错峰折扣”，链接本地商家并探索抽佣/广告变现模式。
